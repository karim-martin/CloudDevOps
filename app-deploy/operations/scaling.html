Scaling

Configure automatic scaling with Horizontal Pod Autoscaler (HPA) for pods and Cluster Autoscaler for GKE nodes.

Step 1: Define Resource Requests

HPA requires resource requests to calculate utilization percentages.

# k8s/backend/deployment.yaml (resources section)
spec:
  containers:
  - name: api
    image: us-central1-docker.pkg.dev/PROJECT_ID/app-repo/backend:latest
    resources:
      requests:
        memory: "128Mi"    # HPA uses this for memory %
        cpu: "100m"        # HPA uses this for CPU %
      limits:
        memory: "512Mi"
        cpu: "500m"

Step 2: HPA with CPU Scaling

# k8s/backend/hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: dotnet-api-hpa
  namespace: app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: dotnet-api
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

HPA with Multiple Metrics

# k8s/backend/hpa-advanced.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: dotnet-api-hpa
  namespace: app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: dotnet-api
  minReplicas: 3
  maxReplicas: 20
  metrics:
  # Scale on CPU utilization
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  # Scale on memory utilization
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # Wait 5 min before scaling down
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60              # Remove 10% of pods per minute max
    scaleUp:
      stabilizationWindowSeconds: 0    # Scale up immediately
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15              # Double pods every 15 seconds max
      - type: Pods
        value: 4
        periodSeconds: 15              # Or add 4 pods every 15 seconds

Apply HPA

# Apply HPA configurations
kubectl apply -f k8s/backend/hpa.yaml
kubectl apply -f k8s/frontend/hpa.yaml

# Verify HPA
kubectl get hpa -n app

# Watch HPA status
kubectl get hpa -n app -w

# Describe HPA for details
kubectl describe hpa dotnet-api-hpa -n app

Cluster Autoscaler

GKE Cluster Autoscaler automatically adds or removes nodes based on pod scheduling needs.

# Enable Cluster Autoscaler on existing node pool
gcloud container clusters update app-cluster \
    --region=us-central1 \
    --enable-autoscaling \
    --min-nodes=1 \
    --max-nodes=10 \
    --node-pool=default-pool

# Or configure per node pool
gcloud container node-pools update default-pool \
    --cluster=app-cluster \
    --region=us-central1 \
    --enable-autoscaling \
    --min-nodes=1 \
    --max-nodes=10

# Verify autoscaling config
gcloud container clusters describe app-cluster \
    --region=us-central1 \
    --format='yaml(nodePools[].autoscaling)'

Autoscaler Profile

# Set autoscaler profile (cluster level)
gcloud container clusters update app-cluster \
    --region=us-central1 \
    --autoscaling-profile=optimize-utilization

# Profiles:
# - balanced (default): Balance availability and cost
# - optimize-utilization: Prefer smaller cluster, more aggressive scale down

Verify Scaling

# Check HPA status
kubectl get hpa -n app
# NAME              REFERENCE              TARGETS   MINPODS   MAXPODS   REPLICAS
# dotnet-api-hpa    Deployment/dotnet-api  45%/70%   3         10        3

# Check current pod resource usage
kubectl top pods -n app

# Check node resource usage
kubectl top nodes

# View HPA events
kubectl describe hpa dotnet-api-hpa -n app | grep -A 10 Events

# View pending pods (Cluster Autoscaler trigger)
kubectl get pods -n app --field-selector=status.phase=Pending

# View Cluster Autoscaler status
kubectl -n kube-system describe configmap cluster-autoscaler-status

Load Testing

Test scaling with artificial load.

# Deploy load generator
kubectl run load-generator \
    --image=busybox \
    --restart=Never \
    --rm -it -- /bin/sh -c \
    "while true; do wget -q -O- http://dotnet-api.app.svc.cluster.local:8080/health; done"

# Or use hey for HTTP load testing
# Install: go install github.com/rakyll/hey@latest
hey -z 5m -c 100 http://YOUR_INGRESS_IP/api/health

# Watch HPA respond
kubectl get hpa -n app -w

# Watch pods scale
kubectl get pods -n app -w

Manual Scaling

# Scale deployment manually
kubectl scale deployment/dotnet-api --replicas=5 -n app

# Note: HPA will override manual scaling if enabled
# To prevent HPA override, temporarily delete HPA:
kubectl delete hpa dotnet-api-hpa -n app

# Scale nodes manually (if autoscaler disabled)
gcloud container clusters resize app-cluster \
    --node-pool=default-pool \
    --num-nodes=5 \
    --region=us-central1

Scaling Best Practices

PracticeRecommendation
CPU target60-80% (leave headroom for spikes)
Min replicasAt least 2-3 for availability
Max replicasBased on budget and capacity
Scale down stabilization300-600 seconds (avoid flapping)
Resource requestsBased on actual usage (monitor first)
Node pool sizingRight-size VMs for your pod sizes

Resource Requests Required

HPA cannot calculate utilization percentages without resource requests. If you see <unknown> in HPA TARGETS, ensure your deployment has CPU and memory requests defined.

Scale Down Delay

By default, HPA waits 5 minutes (stabilizationWindowSeconds) before scaling down. This prevents rapid scaling oscillations during temporary traffic dips.

Preemptible Nodes

Consider using preemptible/spot nodes for cost savings on non-critical workloads. They're up to 80% cheaper but can be terminated with 30-second notice.
